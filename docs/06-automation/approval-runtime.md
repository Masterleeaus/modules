## 1. Purpose

Define where approvals interrupt automation execution, how pending work is persisted, and how Titan safely resumes or terminates execution after an approval outcome.

## 2. Why it exists

Titan is designed so proposed actions are visible and reviewable before higher-autonomy execution. That means runtime automation cannot assume every valid action is immediately executable. Some actions must stop at a governance boundary, wait for approval, and then either resume, expire, or be denied. Approval runtime exists to make that interruption explicit, durable, and safe.

## 3. Core responsibilities

- pause engine execution at approval boundaries
- persist a resumable runtime state before approval is requested
- attach proposal, approval, denial, and expiry outcomes to the engine run
- ensure post-approval resumption re-enters the engine through a safe checkpoint
- prevent bypassing approvals through UI-only shortcuts or direct job replay

## 4. Boundaries

### In scope

- approval checkpoints in runtime execution
- persisted pending state
- approval expiry handling
- denial handling
- resumption after approval
- audit linkage between proposal and runtime execution

### Out of scope

- general workflow-state modeling beyond runtime interruption
- approval UI design details
- broad policy authoring for governance layers
- customer-facing notification copy unless generated by runtime outcomes

## 5. Architecture

Approval runtime is not a separate business workflow. It is an execution gate inside the automation layer.

A typical pattern is:

1. Engine run reaches a step requiring approval.
2. Engine persists checkpoint state, idempotency key, and pending action context.
3. A proposal or approval record is created and linked to the run.
4. The run moves into waiting state rather than continuing.
5. Outcome arrives:
   - approved: resume from checkpoint
   - denied: terminate or reroute according to policy
   - expired: escalate, cancel, or dead-letter depending on action criticality
6. Every path writes audit records and preserves tenant-scoped context.

This ensures approvals are part of durable runtime state, not just a front-end prompt.

## 6. Contracts

Expected inputs:

- engine run identifier
- tenant/company identifier
- action or proposal identifier
- checkpoint reference
- idempotency key
- policy reason for requiring approval
- approval audience or approver class
- expiry window and escalation rules

Expected outcomes:

- approved and resumable
- denied and terminated
- denied and compensating
- expired and escalated
- expired and dead-lettered
- cancelled because source context changed before approval landed

Required persistent state:

- pending approval status
- runtime checkpoint snapshot
- timestamps for requested, viewed, approved, denied, expired
- who acted and under what authority
- replay/resume eligibility marker

## 7. Runtime behavior

When approval is needed, the engine should never keep executing optimistically in the background.

Instead it should:

- checkpoint before any irreversible side effect
- move to a waiting state
- publish enough context for a human or governing AI layer to make a decision
- lock the run against duplicate continuation
- resume only through the same engine path after approval is recorded

Approval-aware runtime must also handle stale context. If the world changed while waiting—for example, capacity is gone, the customer already paid another way, or the target record was edited—the engine should revalidate conditions before resuming. Approval is permission to continue, not proof that continuation is still correct.

## 8. Failure modes

Common risks:

- approval granted after runtime context became stale
- duplicate resume attempts after approval
- denial ignored because a queued retry kept running
- approval request created without a durable checkpoint
- approvals expiring silently with no escalation or closure

Responses:

- re-check prerequisites on resume
- guard resumption with idempotency and run locks
- cancel queued retries once approval wait begins
- require checkpoint persistence before proposal creation
- define expiry policies that route to escalation or dead-letter explicitly

## 9. Dependencies

Upstream:

- triggers and lifecycle transitions
- governance/policy checks
- signal validation and proposal creation
- idempotency and checkpoint storage

Downstream:

- escalation engine for aged approvals
- recovery engine for interrupted approvals
- dead-letter queues for terminal approval failures
- audit and observability
- future Academy/review surfaces and operator dashboards

## 11. Implementation notes

The user’s operating model explicitly places pending Titan actions into a visible review layer before execution, with approved actions executing and denied actions becoming refinement signals. That means approval runtime must be treated as a first-class execution state, not a UI convenience.
